Warum ich meine Websites jetzt in rohem HTML baue (und nicht mehr zurück will)

Oder: Wie ich lernte, WordPress zu verlassen und Jamstack zu lieben

2. November 2025 · 5 Min. Lesezeit

Ich mach jetzt was, das auf den ersten Blick nach Zeitreise aussieht: Ich schreibe meine Websites wieder in purem HTML und CSS. Ja, du hast richtig gelesen. HTML. Wie 1997. Nur dass es 2025 ist und ich alle modernen Tools kenne.

Und nein, das ist keine Nostalgie. Das ist die smarteste Entscheidung, die ich dieses Jahr getroffen habe.

Der Moment der Erkenntnis

Neulich hab ich eine Landing Page gebaut. AIrrenanstalt, crazy-techy Design, Neon-Animationen, das volle Programm. Keine Datenbank, kein Backend, kein CMS. Nur HTML, CSS, und ein bisschen JavaScript.

Upload zu GitHub Pages. Fertig. Live. Kostenlos. In 2 Minuten.

Und dann der Realitätscheck: Die Seite lädt in 0,3 Sekunden. Auf jedem Device. Von jedem Ort der Welt. Ohne dass ich einen Server konfigurieren muss. Ohne dass mich irgendjemand hacken kann.

Ich saß da und dachte: "Fuck. Das ist ja... elegant."

Was ich die letzten 15 Jahre gemacht habe

WordPress. Drupal. Joomla. Typo3 (ja, ich bin alt). Immer das gleiche Spiel:

  • Server mieten (20-100€/Monat)
  • PHP konfigurieren
  • MySQL einrichten
  • CMS installieren
  • Plugins suchen (und hoffen, dass sie nicht kaputt sind)
  • Updates machen (und hoffen, dass nichts bricht)
  • Gehackt werden (weil irgendein Plugin Müll war)
  • Performance-Optimierung (weil 3 Sekunden Ladezeit normal sind)
  • Nächster Hack
  • Repeat

Und das Beste: Für 90% meiner Projekte brauchte ich diese ganze Komplexität GAR NICHT.

Das Jamstack-Paradox

Hier ist der Mind-Fuck: Static Sites sind nicht "zurück zu den Anfängen". Sie sind die Zukunft, die aus der Vergangenheit gelernt hat.

Was wir in den 2000ern dachten:
"Dynamisch ist besser! Server-side rendering! PHP für alles! Datenbanken überall!"

Was wir heute wissen:
Die meisten Websites brauchen keine Datenbank. Sie brauchen HTML, das schnell ausgeliefert wird.

Jamstack nimmt das Beste aus beiden Welten:

  • Die Geschwindigkeit von Static (1990s)
  • Die Flexibilität von APIs (2020s)
  • Die Developer Experience von Git (heute)

Was sich konkret ändert

Performance: Von 3 Sekunden zu 0,3 Sekunden. Das ist keine Optimierung. Das ist ein anderes Universum.

Security: Kein PHP = keine PHP-Exploits. Keine Datenbank = keine SQL-Injection. Keine Admin-Logins = kein Brute-Force. Es gibt buchstäblich nichts zum Hacken.

Kosten: Von 50€/Monat zu 0€. GitHub Pages, Netlify, Vercel - alle kostenlos für kleine bis mittlere Projekte.

Skalierung: Meine WordPress-Seite ist bei 100 gleichzeitigen Besuchern tot. Meine Static Site? Verkraftet eine Million. Via CDN. Automatisch.

Wartung: Keine Plugin-Updates mehr. Keine PHP-Version-Upgrades. Keine MySQL-Patches. Einfach... nichts.

Was ich verliere (und warum es mir egal ist)

Klar, ich verliere ein paar Dinge:

  • Kein Admin-Panel (schreibe ich halt Code - dauert nicht länger)
  • Keine User-Logins (brauch ich in 95% der Fälle eh nicht)
  • Keine dynamischen Kommentare (Disqus/GitHub Issues funktionieren)

Aber weißt du was? Für die meisten Projekte sind das Nicht-Probleme. Landing Pages, Portfolios, Blogs, Dokumentationen, Marketing-Sites - die brauchen keinen dynamischen Backend-Kram.

Und wenn ich's doch mal brauche? APIs. Stripe für Payments. Netlify Forms für Kontaktformulare. Contentful für Content Management. Alles da, alles besser als selbst zu bauen.

Der Workflow ist... schön

Hier ist mein neuer Workflow:

1. Code schreiben (HTML/CSS/JS)
2. git push
3. Automatisch live

That's it. Keine FTP-Uploads. Keine Server-SSH. Kein "Ich hoffe mal, dass das funktioniert."

Und wenn was kaputt geht? git revert. Done. Zurück zur letzten funktionierenden Version in 10 Sekunden.

Warum jetzt?

Zwei Gründe:

Erstens: Die Tools sind endlich da. GitHub Pages, Netlify, Vercel haben Jamstack mainstream gemacht. Früher war Static-Hosting ein Pain. Heute ist es einfacher als WordPress.

Zweitens: Ich hab keine Lust mehr auf Bullshit. Ich will Websites bauen, nicht Server administrieren. Ich will Code schreiben, nicht Plugin-Konflikte debuggen.

Was das bedeutet

Ich bin kein Purist. Wenn ein Projekt wirklich einen dynamischen Backend braucht, baue ich einen. Aber ich frage mich jetzt zuerst: "Kann ich das Static machen?"

Und die Antwort ist öfter "Ja" als ich dachte.

Nike macht's so. Figma macht's so. Smashing Magazine macht's so.

Und ich jetzt auch.

Der Test

Probier's aus. Nimm ein Projekt, das du in WordPress bauen würdest. Bau es in HTML. Upload zu GitHub Pages.

Und dann vergleich:

  • Ladezeit
  • Sicherheit
  • Kosten
  • Wartungsaufwand

Ich wette, du kommst zum gleichen Schluss wie ich.

TL;DR: Ich baue jetzt Static Sites statt WordPress, weil sie schneller, sicherer, billiger und wartungsärmer sind. Das ist nicht Retro. Das ist 2025. Und es fühlt sich verdammt gut an.

Mehr über meinen Workflow und konkrete Projekte gibt's auf meinem GitHub. Ja, ich bin jetzt einer von denen.
